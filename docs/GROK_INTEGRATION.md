# Grok Integration

This document records the Grok SSO account flow used by sub2api and the
behavior exposed through the OpenAI-compatible API.

## Account Credentials

Grok accounts use `platform=grok` and `type=oauth`.

Required credential:

- `sso`: the Grok `sso` session cookie value.

Optional credentials:

- `user_agent`: browser User-Agent used when calling Grok.
- `extra_cookies`: extra cookie string, for example `cf_clearance=...`.

`User-Agent` and `Extra Cookies` can stay empty for normal SSO sessions. They
are only needed when the upstream request requires the same browser fingerprint
or additional anti-bot cookies.

## Models

The Grok gateway exposes these model IDs:

- `grok-4.20-fast`
- `grok-4.20-auto`
- `grok-4.20-expert`
- `grok-imagine-image-lite`
- `grok-imagine-image`
- `grok-imagine-image-pro`
- `grok-imagine-video`

Text models use Grok app-chat with `modeId` mapped from the requested model:

- `grok-4.20-fast`: `fast`
- `grok-4.20-expert`: `expert`
- other text models: `auto`

## Text Chat

External clients use the existing OpenAI-compatible chat endpoint:

```bash
curl "$BASE_URL/v1/chat/completions" \
  -H "Authorization: Bearer $SUB2API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "grok-4.20-fast",
    "messages": [{"role": "user", "content": "hi"}]
  }'
```

Internally, sub2api calls:

```text
POST https://grok.com/rest/app-chat/conversations/new
```

The request is streamed and converted back to OpenAI chat output.

## Lite Images

`grok-imagine-image-lite` does not use WS imagine. It follows the Grok web
lite image path and calls the app-chat API:

```text
POST https://grok.com/rest/app-chat/conversations/new
```

This is the same REST streaming API family used by Grok text chat. The request
is treated as an image-capable chat message by enabling the image generation
flags in the app-chat payload. It does not connect to
`wss://grok.com/ws/imagine/listen`, does not create an imagine post, and does
not call the share/create-link flow.

External clients still use the OpenAI-compatible image endpoint:

```bash
curl "$BASE_URL/v1/images/generations" \
  -H "Authorization: Bearer $SUB2API_KEY" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "grok-imagine-image-lite",
    "prompt": "a clean product photo of a glass teapot",
    "n": 1
  }'
```

The upstream payload is built like:

```json
{
  "message": "Drawing: <prompt>",
  "modeId": "fast",
  "enableImageGeneration": true,
  "enableImageStreaming": true,
  "imageGenerationCount": 2,
  "temporary": true,
  "returnImageBytes": false
}
```

sub2api selects this API path only when either the requested model or the
account model mapping resolves to `grok-imagine-image-lite`. Other image models
continue to use the WS imagine path.

The app-chat stream returns image events in:

```text
result.response.cardAttachment.jsonData.image_chunk
```

sub2api reads:

- `imageUrl`
- `imageUuid`
- `progress`
- `moderated`

Only final events with `progress == 100` and a non-empty `imageUrl` are returned.
Relative asset paths are normalized to:

```text
https://assets.grok.com/<path>
```

Current upstream behavior starts two lite image generations because the web API
uses `imageGenerationCount=2`. sub2api caps the requested count to two and
returns only the number requested by the caller.

Because the upstream lite path still spends two image slots per request, the
recommended local budget policy uses `cost_per_request=2` for `image_lite`.

## Normal And Pro Images

`grok-imagine-image` and `grok-imagine-image-pro` use the WS imagine path.
The upstream task starts a fixed slot batch, so even if an API caller sends
`n=1`, Grok can still consume the upstream task capacity for the whole batch.

Returned URLs are public Grok asset URLs. sub2api does not proxy image bytes.

## Video

`grok-imagine-video` uses the WS imagine video path. After the upstream video
task finishes, sub2api calls Grok's media post `create-link` endpoint to create
the public asset before returning a public URL. If this call fails, sub2api
returns an upstream error instead of returning a guessed URL that may later
resolve to `NoSuchKey`.

The returned video URL is the public share URL generated by Grok. For browser
playback, Grok's public URL can usually be opened with:

```text
?dl=0
```

For direct download, Grok's share page uses:

```text
?dl=1
```

sub2api returns the public URL and leaves download handling to the API caller.

## Billing Defaults

Grok web models use local fallback prices so successful requests are not
recorded as zero-cost usage when no channel pricing is configured:

- text (`grok-4.20-fast`, `grok-4.20-auto`, `grok-4.20-expert`): $1.25 / 1M
  input tokens, $0.20 / 1M cached input tokens, $2.50 / 1M output tokens.
- lite/normal images: $0.02 per returned image.
- pro images: $0.05 for 1K, $0.07 for 2K, $0.14 for 4K per returned image.
- video: $0.05 per generated second; missing duration defaults to 6 seconds.

These defaults are only the billing fallback. Channel model pricing can still
override them per group/model.

## Scheduling And Failure Feedback

Grok account selection is load-aware:

- filters disabled or temporarily unschedulable accounts,
- filters accounts that do not support the requested model,
- filters accounts under per-model rate-limit cooldown,
- prefers lower priority value, then lower runtime load,
- round-robins accounts inside the same priority/load bucket per group and
  Grok scope,
- uses stricter concurrency defaults for media models.

Default per-account concurrency caps:

- text: 2
- lite image: 2
- normal/pro image: 1
- video: 1

Upstream failures feed back into scheduling:

- invalid SSO/session: mark account error and pause for 24 hours,
- anti-bot rejection: pause for 30 minutes,
- rate limit: set model cooldown for 2 hours,
- transient EOF/disconnect: pause for 2 minutes.

## Local Budget Policy

Grok media quotas are not fully exposed by the public web endpoints, so sub2api
supports an optional local budget policy on each account:

```json
{
  "grok_budget_policy": {
    "image_lite": { "limit": 100, "window_seconds": 7200, "cost_per_request": 2 },
    "image_normal": { "limit": 60, "window_seconds": 7200, "cost_per_request": 6 },
    "image_pro": { "limit": 40, "window_seconds": 7200, "cost_per_request": 4 },
    "video": { "limit": 10, "window_seconds": 86400, "cost_per_request": 1 }
  }
}
```

If no local policy is configured, Grok scheduling fails open and only relies on
runtime errors and cooldowns. If Redis budget accounting is unavailable,
requests also fail open instead of blocking all Grok traffic.
